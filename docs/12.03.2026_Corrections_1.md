# 12.03.2026 Corrections 1 — Methodological Hardening

## Overview

This document details the methodological hardening fixes applied to CalibrateMe to address reviewer concerns about threshold transparency, analytical reproducibility, and methodological honesty in the UI.

---

## Task 1: Centralize All Analytical Thresholds

### Problem
18 analytical thresholds (e.g., `0.10` for good calibration ECE, `0.9` for mastery K*, `0.5` for automatization ratio) were hardcoded across 4 analytics modules. This made threshold sensitivity analysis impossible and raised reproducibility concerns.

### Solution
Created a single source of truth: `src/config/analyticsThresholds.ts`

```typescript
export const ANALYTICS_THRESHOLDS = {
  session_break_minutes: 10,
  regression_threshold_pp: 15,
  breakout_threshold_pp: 15,
  good_calibration_ece: 0.10,
  phase_novice_k_star: 0.40,
  phase_mastered_k_star: 0.90,
  phase_plateau_gain_rate: 0.005,
  phase_min_sessions_for_plateau: 5,
  automatization_type1_ratio: 0.50,
  high_effort_rt_multiplier: 1.5,
  alpha_high: 0.25,
  alpha_low: 0.15,
  lambda_low: 0.07,
  lambda_high: 0.12,
  beta_well_calibrated: 0.08,
  beta_severe: 0.18,
  classification_overconfident: 0.10,
  classification_underconfident: -0.10,
} as const;
```

### Files Modified
- **Created**: `src/config/analyticsThresholds.ts` — Central config with `as const` + exported type
- **Modified**: `src/features/analytics/longitudinalAnalysis.ts` — Added optional `thresholds?: Partial<AnalyticsThresholds>` parameter to `analyzeLongitudinal()`, replaced hardcoded values in `analyzeLearningVelocity()` and `analyzeSessionQuality()` with threshold references
- **Modified**: `src/features/analytics/patternAnalysis.ts` — Added optional `thresholds?` parameter to `analyzePatterns()`, replaced `0.5` (automatization ratio) and `1.5` (high effort multiplier) with threshold values
- **Modified**: `src/features/analytics/learnerInsights.ts` — Added optional `thresholds?` parameter to `analyzeLearnerInsights()`, replaced 10 hardcoded values across `interpretParams()`, `detectStrengthsWeaknesses()`, and `classifyArchetype()`
- **Modified**: `src/features/analytics/components/LearnerClassification.tsx` — Imported `ANALYTICS_THRESHOLDS`, replaced hardcoded `0.1`/`-0.1` classification thresholds and `10 * 60 * 1000` session gap

### Design
All three analytics modules now accept `thresholds?: Partial<AnalyticsThresholds>` and merge with defaults via `{ ...ANALYTICS_THRESHOLDS, ...thresholds }`. This preserves backward compatibility (no argument = default behavior) while enabling threshold sensitivity sweeps.

## Task 2: Threshold Sensitivity Sweep

### Problem
Reviewers questioned whether analytical conclusions (phase labels, archetype classifications, calibration quality ratings) were robust to threshold choices or artifacts of specific cut-off values.

### Solution
Created `src/simulation/thresholdSensitivity.ts` — a module that re-runs analytical interpretation (not simulation) with varied thresholds on existing simulation results.

### Key Design Decisions
- **No re-simulation needed**: The module takes pre-computed `SimulationResults` and re-derives labels with different threshold overrides. This is fast (seconds, not minutes).
- **7 parameter sweeps, 25 total values**: Covers session break minutes, regression threshold, good calibration ECE, automatization ratio, high effort RT multiplier, phase mastery K*, and beta well-calibrated threshold.
- **4 conclusions tracked per variation**: learning phase, calibration quality, archetype, automatization session.

### Files Created
- `src/simulation/thresholdSensitivity.ts` — Core module with `runThresholdSensitivity()`, `thresholdSensitivityToCSV()`, `DEFAULT_THRESHOLD_SWEEPS`
- `scripts/runThresholdSensitivity.ts` — CLI script: generates representative simulations, runs sweeps, exports CSV, prints per-parameter sensitivity breakdown

### Files Modified
- `scripts/generateLatexTables.ts` — Added `generateThresholdSensitivityTable()` function

## Task 3: Matched Scaffold Comparison (Paired Within-Subject Design)

### Problem
The existing ablation study compared conditions across different profiles but didn't isolate the scaffolding effect with proper statistical controls. A paired within-subject design (same profile, same seed, with vs. without scaffolding) was needed.

### Solution
Created `src/simulation/matchedScaffoldAnalysis.ts` — a dedicated module implementing:
- **Paired design**: Same profile + same random seed, scaffolding on vs. off
- **Custom paired t-test**: Implemented via incomplete beta function with Lanczos gamma approximation (no external stats library dependency)
- **Paired Cohen's d**: Effect size computed on the difference scores
- **Per-profile results**: Retention, ECE, and mastery differences with 95% CIs

### Files Created
- `src/simulation/matchedScaffoldAnalysis.ts` — Types `MatchedPairResult`, `MatchedScaffoldReport`; functions `runMatchedScaffoldComparison()`, `matchedScaffoldToCSV()`

### Files Modified
- `scripts/runAllAnalyses.ts` — Added Step 5 (matched scaffold comparison, 9 profiles x 30 seeds x 2 conditions), updated files_generated list, added `experiment_config_version` and `experiment_config_path` to summary JSON
- `scripts/generateLatexTables.ts` — Added `generateMatchedScaffoldTable()` function

## Task 4: UI "Honesty Labels" (Heuristic Tooltips)

### Problem
The UI presents heuristic-derived labels (e.g., "Overconfident", "mastered", "automatic") as if they were ground truth. Methodological transparency requires disclosing that these are researcher-chosen heuristic thresholds.

### Solution
Created a reusable `HeuristicTooltip` React component and added it to 7 locations across 5 UI components.

### Files Created
- `src/components/HeuristicTooltip.tsx` — Reusable component with hover/click tooltip, `role="note"`, `aria-label`, click-outside dismiss

### Files Modified (tooltip additions)
1. **`LearnerClassification.tsx`** — Classification badge ("Overconfident"/"Underconfident"/"Well-Calibrated") now has tooltip explaining β̂ threshold cut-offs
2. **`FinalReport.tsx`** — "Learner Archetype" label has tooltip explaining it's heuristic-based; "Current Phase" label has tooltip explaining K* threshold cut-offs
3. **`ECEMeter.tsx`** — ECE severity label ("Well Calibrated"/"Moderate"/"Needs Improvement") has tooltip explaining threshold conventions
4. **`SessionHistory.tsx`** — "Calibration Trend" heading has tooltip explaining session-splitting heuristic and bin count
5. **`SessionSummary.tsx`** — "Response Patterns" heading has tooltip explaining dual-process classification heuristic; added subtitle "Estimated from timing and confidence patterns"

### CSS Added
- `src/App.css` — `.heuristic-tooltip`, `.heuristic-tooltip__icon`, `.heuristic-tooltip__popup` with dark mode variant, `.heuristic-subtitle`

## Task 5: Frozen Reproducibility Config

### Problem
Experiment parameters were scattered across script files. For IEEE reproducibility, all parameters should be documented in a single frozen configuration file.

### Solution
Created `config/experiment_config.json` (version 1.0.0) containing:
- Ablation config: 30 seeds, 6 conditions, 9 core + 6 extended profiles
- Sensitivity config: 10 seeds, 3 profiles, 5 parameters with sweep values
- Delta sweep config: 15 seeds, 8 delta values
- Matched scaffold config: 30 seeds
- Simulation defaults: 100 items, 30 sessions, 20 items/session
- All 18 analytics thresholds
- Threshold sensitivity sweep values (7 parameters)

### Files Created
- `config/experiment_config.json`

### Files Modified
- `README.md` — Added "Reproducing Results" section with instructions; updated "Generated Results" listing with new CSV/LaTeX files
