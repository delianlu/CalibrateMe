## Fix 1: Fair Crash Detection Criteria

### Problem

The CrammerCrashChart used **asymmetric** crash detection:
- SM-2: K̂ < 0.5 AND K* < 0.5 (reactive, waits for actual crash)
- CalibrateMe: β̂ > 0.15 (proactive early warning)

These are not comparable events. The comparison appeared rigged in CalibrateMe's favor.

### Solution

Replaced with a **common ground-truth crash criterion** and **warning lead time** metric:

1. **Ground-truth crash**: First session where K* drops below 0.50 after being ≥ 0.50
2. **Warning signals** (same type of signal for both):
   - SM-2: Session where accuracy drops by >5% from previous session
   - CalibrateMe: Session where β̂ > 0.15 OR accuracy drops by >5%
3. **Lead time**: Sessions between warning and ground-truth crash (positive = early detection)

### UI Changes

- Added K*=0.50 reference line on knowledge tracking chart
- Added bold "Crash" reference line at ground-truth crash session
- Changed "detects" labels to "warns" labels
- Summary table now shows:
  - Ground-Truth Crash row (shared, spans both columns)
  - Warning Signal row (per-scheduler)
  - Warning Lead Time row with directional comparison text

### LaTeX Table Update

Added `Ground-Truth Crash`, `Warning Signal (session)`, and `Warning Lead Time` rows replacing the old single `Crash Detection` row.

## Fix 2: Randomize Item Types

### Problem

Items were assigned types by alternation (`i % 2 === 0 → VOCABULARY`), confounding domain with index, recency, and scheduler interaction patterns.

### Solution

In `simulationEngine.ts`, when `enable_domain_split` is true, item types are now reassigned using the seeded RNG **after** profile creation but **before** the session loop:

```typescript
if (domainSplitEnabled) {
  for (const item of learner.items) {
    item.item_type = random.random() < 0.5 ? ItemType.VOCABULARY : ItemType.GRAMMAR;
  }
}
```

**Key properties:**
- Approximately 50/50 split (not exact, which is correct for randomization)
- Reproducible with the same seed
- Different seeds produce different assignments
- Only affects domain-split simulations; non-domain-split profiles retain the original alternating assignment from `createLearnerProfile` (which is unused in that mode anyway)

Added `ItemType` to the imports in `simulationEngine.ts`.

## Fix 3: Domain β̂ Minimum Responses (3 → 5)

### Problem

`updateDomainBetaHat()` required only 3 responses per domain before updating β̂_domain. With EMA-based estimation, 3 responses is too noisy to reliably influence scheduling intervals.

### Solution

Changed threshold from 3 to 5 in both branches of `updateDomainBetaHat()`:

```typescript
// Before
if (vocabResponses.length >= 3) { ... }
if (grammarResponses.length >= 3) { ... }

// After — with documentation
// Minimum 5 responses per domain provides a more stable EMA estimate
// before influencing scheduling (with ~10 items/domain per session,
// this threshold is typically met by session 1)
if (vocabResponses.length >= 5) { ... }
if (grammarResponses.length >= 5) { ... }
```

**Impact:** With 20 items per session and ~50% per domain (now randomized), each domain receives ~10 responses per session. The threshold of 5 is met within the first session, so domain β̂ still begins updating early — just not from the first 3 responses.
