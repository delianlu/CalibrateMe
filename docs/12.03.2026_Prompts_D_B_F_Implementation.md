## Prompt D: Crammer's Crash Stress Test

### Motivation

The Crammer profile models a learner who crams material (achieving short-term fluency) but forgets rapidly and maintains high, unwarranted confidence. The key research question: **Does CalibrateMe detect the crammer's crash faster than SM-2?**

SM-2 trusts learner confidence directly when computing quality ratings, so an overconfident crammer will receive inflated intervals. CalibrateMe tracks β̂ (calibration bias) and shortens intervals when overconfidence is detected via the `e^{-2β̂}` adjustment factor.

### Profile Parameters

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| β* (calibration bias) | +0.30 | Extreme overconfidence |
| λ (forgetting rate) | 0.25 | Very fast forgetting (2.5× median) |
| α (learning rate) | 0.10 | Slow actual learning |
| initial_k_star | 0.50 | Crammed starting knowledge |
| rt_base | 2.0s | Faster responses (surface fluency) |
| rt_gamma | 1.5 | Weaker knowledge-RT relationship |
| confidence_noise_std | 0.08 | Lower noise (consistently overconfident) |

## Prompt B: Domain-Split Metacognition

### Motivation

Research in metacognition suggests that self-monitoring accuracy can vary across content domains — a learner might be well-calibrated on vocabulary but overconfident on grammar. A single global β̂ averages over these differences, potentially missing domain-specific miscalibration that could inform better scheduling.

The key research question: **Does domain-specific calibration tracking improve scheduling for asymmetrically miscalibrated learners?**

### Architecture

```
                  ┌─────────────────┐
                  │  Response Gen   │
                  │  Uses β*_domain │
                  └────────┬────────┘
                           │
              ┌────────────▼────────────┐
              │   Dual-Process Classify  │
              └────────────┬────────────┘
                           │
         ┌─────────────────▼─────────────────┐
         │         BKT Belief Update          │
         │  updateBetaHat() → global β̂       │
         │  updateDomainBetaHat() → β̂_v, β̂_g │
         └─────────────────┬─────────────────┘
                           │
              ┌────────────▼────────────┐
              │   Scheduler: uses       │
              │   β̂_domain for item     │
              │   interval adjustment   │
              └─────────────────────────┘
```

### Type Extensions (`src/types/index.ts`)

New types added:

```typescript
enum ItemType { VOCABULARY = 'VOCABULARY', GRAMMAR = 'GRAMMAR' }

interface DomainCalibration {
  beta_hat_vocab: number;
  beta_hat_grammar: number;
}
```

Modified interfaces:
- `Item`: Added optional `item_type?: ItemType`
- `Response`: Added optional `item_type?: ItemType`
- `SystemBelief`: Added optional `domain_calibration?: DomainCalibration`
- `SimulationConfig`: Added `enable_domain_split: boolean` (default: `false`)
- `SimulationResults`: Added optional `beta_hat_vocab_trajectory` and `beta_hat_grammar_trajectory`
- `SessionData`: Added optional `beta_hat_vocab` and `beta_hat_grammar`

All fields are optional to maintain backward compatibility.

### Belief Update Engine (`src/bkt/beliefUpdateEngine.ts`)

New function `updateDomainBetaHat()`:

```typescript
function updateDomainBetaHat(
  responses: Response[],
  current: DomainCalibration,
  learning_rate: number = 0.1
): DomainCalibration
```

- Filters responses by `item_type` (vocabulary vs grammar)
- Requires minimum 3 responses per domain before updating (avoids noisy early estimates)
- Applies independent EMA: `β̂_domain = β̂_domain + lr × (observed - β̂_domain)`
- Returns updated `DomainCalibration` object

### Response Generation (`src/simulation/responseGenerator.ts`)

`generateResponse()` now accepts an optional `domainBetaStar` parameter:

```typescript
function generateResponse(
  item: Item,
  learner_state: TrueLearnerState,
  config: SimulationConfig,
  random: SeededRandom,
  domainBetaStar?: { vocab: number; grammar: number }
): Response
```

When domain-split is enabled and the item has a type, the function uses the domain-specific β* instead of the global `learner_state.beta_star` for confidence generation. This models learners whose overconfidence differs by content domain.

### Calibration-Aware Scheduler (`src/scheduler/calibrationAwareScheduler.ts`)

`CalibrateMeScheduler.processResponse()` now checks for domain calibration:

```typescript
if (belief.domain_calibration && response.item_type) {
  const domainBetaHat = response.item_type === ItemType.VOCABULARY
    ? belief.domain_calibration.beta_hat_vocab
    : belief.domain_calibration.beta_hat_grammar;
  effectiveBelief = { ...belief, beta_hat: domainBetaHat };
}
```

This means vocabulary items are scheduled using β̂_vocab and grammar items using β̂_grammar, producing different interval adjustments per domain. An overconfident-on-vocab learner gets shorter vocab intervals without affecting grammar scheduling.

### Simulation Engine (`src/simulation/simulationEngine.ts`)

Changes to `runSimulation()`:
1. Initializes `domain_calibration` in system belief when `enable_domain_split` is true
2. Constructs `domainBetaStar` from profile params if `beta_star_vocab` and `beta_star_grammar` are defined
3. Passes `domainBetaStar` to `generateResponse()` for domain-specific confidence generation
4. Calls `updateDomainBetaHat()` after the global β̂ update when domain split is enabled
5. Tracks domain β̂ trajectories and records per-session domain β̂ values

## Design Decisions

### 1. Profile-Level Config Overrides vs Separate Config Objects

**Decision:** Optional fields on `LearnerProfileParams` with spread-merge in `runSimulation()`.

**Rationale:** Avoids proliferating SimulationConfig variants. The Crammer profile needs different `rt_base` and `confidence_noise_std`, but these are profile-specific behavioral characteristics, not experiment-level settings. The optional field approach keeps the API clean and backward-compatible.

### 2. Domain Split as Optional Feature Flag

**Decision:** `enable_domain_split: boolean` on SimulationConfig, defaulting to `false`.

**Rationale:** Domain-split adds complexity that only benefits asymmetrically miscalibrated learners. Making it opt-in means the core 9 profiles behave identically to before, and the feature can be toggled for research comparisons.

### 3. Minimum 3 Responses for Domain β̂ Update

**Decision:** `updateDomainBetaHat()` requires 3+ responses per domain before updating.

**Rationale:** With only 1-2 responses, the observed β̂ is dominated by noise. Waiting for 3 responses gives a more stable signal before adjusting scheduling intervals.

### 4. Pre-computed Data for SplitScreenDemo

**Decision:** Run both simulations in `useMemo` on mount, then animate through pre-computed snapshots.

**Rationale:** Running simulations during animation would cause jank. Pre-computing ~100ms of work once means smooth 60fps playback during the presentation.
